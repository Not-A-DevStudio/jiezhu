## 概述

Jiezhu 官网的路由看起来不复杂，实际上也确实不繁琐。总的来说，我们的路由设计可以拆成三层：

1. 站点级页面路由：用来切换首页、Demo 页和博客页。
2. 首页内部的区块跳转：用来在同一页面里滚动到不同 section。
3. 博客页内部的文章切换：控制当前显示哪一篇文章。

这样做的好处是，用户看到的导航非常直接，开发时也能把“页面切换”和“页面内定位”分开处理，不会把所有交互都硬塞进一套路由系统里。

## HashRouter

在针对“页面切换”做路由技术选型时，我们有2种常见的方案：HashRouter 和 BrowserRouter。最终我们选择了 HashRouter，主要是因为它在静态托管时不需要额外配置服务器重写规则，刷新页面也不会因为服务端找不到真实路径而 404。这对于一个部署在静态站点上的前端项目来说，是一个非常稳妥且省事的方案。毕竟我们的网站是托管在 Github Pages 上的，我们对服务端其实没多少（根本没有）控制权。

## 锚点

一开始我们的官网的顶部导航实际使用的是 `/#features`、`/#start` 等锚点链接，后来我们发现这样做有个问题：如果用户当前不在首页，点击导航会直接跳转到 /#features，但由于页面还没有渲染完成，目标区块还没有挂载，这时就会导致滚动失败，用户体验很差。还有一个就是，事实上 Jiezhu 项目的首页并不需要“一个 url 直接跳转到对应 section ”的功能，于是秉持“**Do less**”原则。我们把顶部导航的交互逻辑改成了：如果当前就在首页，点击导航会直接 `scrollIntoView`；如果当前不在首页，先 `navigate("/")` 回到首页，再等页面渲染完成后滚到目标区块。这样就把“返回首页”与“定位 section”拆成了两个步骤，避免了之前遇到的滚动问题。

## Query

在博客页内部，我们需要控制当前显示哪一篇文章。最初因为开发经验不足，我根本没有考虑到这个问题。好在之前很长一段时间内，想要访问博客，都需要用户手动在 Header 导航到博客页，所以那段时间没出问题。

于是在优化博客页入口，在首页展示时，我们发现博客没法正常定位到对应文章了。于是我们决定在 URL 上使用 Query 参数来标记当前文章，比如 `?id=placeholder`。这样用户在首页点击某篇博客的链接时，就会跳转到 `/blog?id=placeholder`，进入博客页后我们就可以根据这个 Query 参数来加载并展示对应的文章了。

> 刚写到这里 Lisa 又给我加需求了，要给博客加一个动态加载。

之前 Jiezhu 官网博客都是直接让 Vite 在构建时把所有博客文章都直接硬编码进 JS 里，每次打包都有警告。现在我们改成了在博客页加载时才去请求博客文章列表和内容，这样首页的加载速度就不会受到博客文章数量的影响了。

然后在加上动态加载之后，网站直接在开发环境白屏了？

紧急打开开发者工具，发现是请求博客文章时的 URL 错误了。

```text
Uncaught TypeError: Failed to construct 'URL': Invalid URL
    at blog.ts:1:20
```

后来才发现是我在构建 URL 时硬编码了 baseURL，导致在开发环境下请求博客文章时 URL 不对。这个问题的解决方案也很简单，把 baseURL 改成相对路径就好。

## 小结

回顾 Jiezhu 官网的路由设计，我们最终形成了三层清晰的技术选型思路：

1. **站点级页面路由 —— HashRouter**  
   选择 HashRouter 而非 BrowserRouter，核心原因是Jiezhu官网部署在静态托管平台 GitHub Pages 上，使用 HashRouter 可以无需配置服务端重写规则，即可避免刷新 404。虽然 URL 中会带有 `#`，但换来了稳定可靠的页面切换体验，属于“稳妥省事”的务实之选。

2. **首页内部区块跳转 —— 锚点 + 条件滚动**  
   放弃纯锚点链接（如 `/#features`），改为交互逻辑判断：若当前在首页则直接 `scrollIntoView`；若不在首页则先跳转首页，待页面渲染完成后再滚动到目标区块。这一方案解决了跨页面滚动失败的问题，也体现了“Do less”原则——不为不需要的功能（如直接通过 URL 访问首页某区块）增加复杂度。

3. **博客页文章切换 —— Query 参数 + 动态加载**  
   使用 Query 参数（如 `?id=xxx`）标记当前文章，配合动态加载机制：仅在进入博客页时才请求文章列表与内容，避免构建时将所有文章打包进首页 JS，从而优化首页加载性能。

整体而言，这套设计没有追求“纯前端路由的极致优雅”，而是根据实际部署环境、~~假想中的~~用户体验痛点（跨页面锚点失效）和性能要求（博客文章膨胀），做了务实的取舍。希望这些思路对类似场景的开发者有所启发。
